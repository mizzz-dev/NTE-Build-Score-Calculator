# `/card`・`/compare` OCR入力補助 展開可否レビュー（Issue #103）

- 作成日: 2026-05-13
- 対象Issue: #103
- 前提: `/score` OCR品質評価で第3・第4サイクル連続2サイクル全KPI達成済み
- 本書の範囲: 展開可否判断・影響整理・次Issue候補整理（**実装なし**）

## 1. 結論（判定）

判定: **条件付き可**

理由:
1. `/score` で2サイクル連続全KPI達成により、展開検討ゲートは通過済み。
2. ただし低スペック端末カテゴリp95が4.1秒で、カテゴリ単体では目標4.0秒に僅差未達。
3. `/card`・`/compare` は `/score` より入力文脈が複雑化し、手動補正前提導線の品質が重要。
4. payload非混入・ランキング/共有互換維持は絶対条件であり、事前の回帰観点固定が必要。

## 2. `/card` OCR入力補助のユースケース整理

1. **ビルドカード作成時の装備情報入力短縮**
   - 装備タイプ/メイン・サブステータスを画像から下書き反映し、カード生成前に手動確定する。
2. **既存手入力の補完**
   - 一部のみOCRで補完し、未認識項目は従来通り手入力する。
3. **複数装備の連続入力時の負荷軽減**
   - 同一セッション内で複数回OCRを実行し、入力時間を短縮する。
4. **失敗時の即時手動fallback**
   - 読み取り失敗時にカード作成フローを止めず、即手動に戻れることを重視する。

## 3. `/compare` OCR入力補助のユースケース整理

1. **比較対象A/Bの入力下書き**
   - 比較対象2セットの装備情報入力をOCRで下書きし、比較実行前に手動確認する。
2. **片側のみOCR利用**
   - 既存データ片側 + 新規OCR片側の混在入力を許容し、運用柔軟性を担保する。
3. **差分確認の事前検証**
   - OCR反映値が比較結果に与える影響を、ユーザーが実行前に確認できる導線を維持する。
4. **比較誤認時の影響最小化**
   - OCR誤認が比較結論を歪めないよう、未確定項目の明示と再編集導線を必須化する。

## 4. `/score` との差分整理

| 観点 | `/score`（現行） | `/card` 展開時 | `/compare` 展開時 | 影響評価 |
|---|---|---|---|---|
| 入力対象 | 単一入力フォーム中心 | カード生成に必要な入力群 | 比較用に2系統入力 | 対象項目数・同時編集数が増える |
| UI導線 | OCR→候補確認→反映 | OCR→カード反映前確認が必要 | OCR→比較実行前の両系統確認が必要 | 誤反映防止導線の強化が必要 |
| 手動補正必要性 | 低信頼度時に必須 | 同等以上に必須（生成物に直結） | 最も必須（比較結果の信頼性に直結） | 未確定可視化が重要 |
| 保存payload影響 | 非混入保証済み | **非混入維持必須** | **非混入維持必須** | 仕様変更不可 |
| ランキング/共有URL影響 | 互換維持済み | 互換維持（影響なし前提） | 互換維持（影響なし前提） | 回帰確認が必要 |
| パフォーマンス影響 | p95=3.8秒（全体） | 実行回数増で体感待機増加リスク | 2系統入力で待機機会増 | 端末別監視強化が必要 |

## 5. セキュリティ・プライバシー・権利面の整理

1. OCRは入力補助であり、最終確定はユーザー手動確認を必須維持する。
2. 画像はサーバー保存しない（メモリ一時処理）方針を維持する。
3. OCR由来メタ情報（confidence等）は保存payload・ランキングpayload・共有URLへ混入させない。
4. 個人情報・アカウント情報が含まれる画像を扱わない注意喚起を継続する。
5. 非公式ファンツール表記を維持し、公式素材利用可否は引き続き「要確認」とする。

## 6. 監視条件・ロールバック条件・段階展開方針

### 6.1 監視条件（必須）
- KPI（名称/値/マッピング/補正率/p95/fallback/誘導率）を `/card` `/compare` 個別で計測。
- 端末性能別p95を個別追跡（特に低スペック端末）。
- 未確定項目残存率と、反映前確認ステップ離脱率を追加監視。
- payload非混入の回帰チェックを各リリースで実施。

### 6.2 ロールバック条件（必須）
- いずれかのKPIが2サイクル連続未達。
- OCR誤反映に起因する重大インシデント（比較結果の誤誘導等）が発生。
- payload混入・共有URL互換破壊・ランキング不整合のいずれかを検知。
- 低スペック端末でp95悪化トレンド（例: 4.3秒超の継続）を確認。

### 6.3 段階展開方針
1. Phase A: `/card` のみ限定導入（feature flag前提）
2. Phase B: `/card` でKPI安定後に `/compare` へ限定導入
3. Phase C: `/card` `/compare` 同時常時化判定
