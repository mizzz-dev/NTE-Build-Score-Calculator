# Issue #113: `/compare` OCR入力補助 要件詳細化・監視設計

- 作成日: 2026-05-14
- 対象Issue: #113
- 関連: #103（条件付き可）, #105（`/card`要件固定）, #109, #111
- 範囲: **要件定義のみ（実装なし）**
- 非対象: `/compare` OCR実装、OCRアルゴリズム変更、UI大幅変更、DB migration、auth/infra/deployment変更、保存payload/ランキング仕様変更

## 0. 前提と目的

`/card` OCR限定導入は Issue #109 / #111 の2サイクル連続で継続条件を達成した。次フェーズとして `/compare` へのOCR入力補助展開可否を安全性観点で詳細化する。本ドキュメントは、`/compare` への実装着手前に責務境界・非混入・比較正当性・監視・ロールバック条件を固定する。

本Repositoryは非公式ファンツールであり、公式素材利用可否は継続して「要確認」とする。

## 1. `/compare` の既存責務整理

`/compare` は以下の責務を持つ。

1. 入力責務
   - 比較対象A/Bの2系統入力を受け付ける（手動入力が正本）。
2. 表示責務
   - A/Bの入力値と比較差分を表示する。
3. 比較計算責務
   - 入力値を比較計算へ渡し、差分結果を算出する。
4. 共有責務
   - 既存共有URL仕様に従って比較状態をエンコードする。
5. 保存/ランキング連携責務
   - 既存payload仕様で保存・ランキング連携を行う。

> OCR導入後も、3-5の責務境界は不変とする。

## 2. `/score`・`/card` から流用する責務 / `/compare` 固有責務

### 2.1 流用可能な共通責務

- OCR結果は画面内一時状態（下書き）に限定し永続化しない。
- 低信頼度候補は自動確定しない（手動確認必須）。
- OCR失敗時は即時手動fallbackへ戻す。
- OCRメタ（confidence/raw text/candidates/matchType）は保存payload・共有URL・ランキングpayloadへ混入させない。
- 画像はサーバー保存せずメモリ一時処理のみ。

### 2.2 `/compare` 固有で分離すべき責務

- A/B 2系統それぞれの下書き状態を分離管理する責務。
- A/Bのどちらか一方のみOCR利用する混在入力を許容する責務。
- 比較実行前に、A/B両方の未確定項目有無を検査する責務。
- 未確認OCR値を比較計算入力に流入させないガード責務。
- A/B取り違え誤反映（cross-side contamination）を防ぐ責務。

## 3. `/compare` OCR入力補助 主ユースケース

1. A/B両系統の入力下書き作成をOCRで補助し、手入力負荷を軽減する。
2. 片側のみOCR、もう片側は手入力/既存値で比較を行う。
3. OCRで認識できた項目のみ先行反映し、未確定項目は手動補完する。
4. OCR失敗時でも比較フローを止めず、即時手動入力へ復帰する。

## 4. A/B 2系統入力におけるOCR反映ルール

1. OCR実行の適用先（A or B）を開始時点で明示し、結果は対象系統にのみ保持する。
2. A/Bそれぞれに `draft` / `confirmed` 状態を持ち、系統間で共有しない。
3. 下書き反映対象は `/score` `/card` と同一（装備タイプ、メイン/サブステータス名・値）に限定する。
4. A/Bの片側に未確定項目が残る場合、その系統は比較計算へ投入不可とする。
5. A/B切替時は未反映下書きを自動移送せず、利用者の明示操作でのみ反映する。

## 5. 下書き扱い・比較前手動確認 必須方針

- OCR結果は常に「下書き（未確定）」として扱う。
- 比較実行前に、A/B各系統で「反映内容確認」を必須化する。
- 最低1回の手動確認アクション（確認チェックまたは明示承認）なしに比較実行を許可しない。
- 比較結果画面へ遷移する条件は「A/Bとも確認済み」かつ「未確定項目0件」とする。

## 6. 低信頼度候補の扱い

- confidence閾値未満（暫定 `< 0.85`）は自動確定禁止。
- マッピングがpartial/similarの項目も未確定扱いを維持。
- 未確定項目は「要確認」表示として残し、空欄または候補選択待ちにする。
- 低信頼度項目が残存する限り、比較実行はブロックする。

## 7. payload/URL/ランキングへのOCRメタ非混入方針

以下は **保存payload・共有URL・ランキングpayloadへ混入禁止**。

- confidence
- raw text
- matchType
- unresolvedReason
- 候補配列（candidates）
- OCR処理時間・端末カテゴリ等の計測メタ

実装時は既存payload builderを再利用し、OCR情報は比較前のUI一時状態に閉じ込める。

## 8. 比較結果正当性（未確認OCR値の混入防止）

比較計算呼び出し前に、以下のガード条件を満たす必要がある。

1. A/B双方で確認済みフラグがtrue。
2. A/B双方で未確定項目数が0。
3. A/B双方で必須入力欠損が0。
4. A/B双方でOCR処理由来のエラー状態が解消済み。

いずれか未達の場合は比較実行不可とし、対象系統の修正導線へ戻す。

## 9. OCR失敗時の手動fallback導線

1. 読み取り失敗時は対象系統のみ失敗表示し、他系統入力は保持する。
2. 即時に手動入力へ切替可能とし、比較フローを中断させない。
3. 再試行は任意とし、強制しない。
4. 失敗後にOCRを使わず比較を完遂できることを受け入れ条件に含める。

## 10. `/compare` OCR導入時の監視KPI

### 10.1 必須KPI

- OCR処理時間 p95（全体）
- OCR処理時間 p95（端末性能別: 低/中/高）
- 2系統入力の誤反映率（A/B取り違え・誤side反映）
- 未確定項目残存率（比較実行直前時点）
- 比較前確認離脱率
- fallback率（OCR失敗/手動切替）
- 補正率（OCR反映後の手動修正割合）

### 10.2 監視運用条件

- 限定公開中は日次集計＋週次レビュー。
- 低性能端末カテゴリは個別閾値監視を必須化。
- `/card` と同一形式で匿名・集計値のみ記録し、生画像は保持しない。

## 11. 回帰観点（非混入・互換・正当性）

1. payload非混入回帰
   - OCRメタが保存payloadへ存在しないこと。
2. 共有URL互換回帰
   - OCR有無でURL仕様差分が発生しないこと。
3. ランキング互換回帰
   - OCR有無でランキングpayload構造差分がないこと。
4. 比較結果正当性回帰
   - 未確認/未確定値が比較結果に混入しないこと。
5. 手動入力互換回帰
   - OCR未使用時の既存比較導線・結果が不変であること。

## 12. 限定公開時ロールバック条件

以下のいずれかを満たす場合、`/compare` OCR導線をロールバック候補とする。

- KPIが2サイクル連続未達。
- 2系統誤反映率が許容閾値を超過し、改善トレンドなし。
- 未確定値混入による比較誤誘導インシデント発生。
- payload混入、共有URL非互換、ランキング不整合の検知。
- 低性能端末p95が悪化トレンド（例: 4.3秒超継続）。

ロールバック手段はfeature flagで`/compare` OCR導線を無効化し、従来の手動入力比較へ戻す。
