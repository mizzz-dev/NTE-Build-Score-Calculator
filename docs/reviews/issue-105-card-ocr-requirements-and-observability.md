# Issue #105: `/card` OCR入力補助 要件詳細化・監視設計

- 作成日: 2026-05-13
- 対象Issue: #105
- 関連: #103（判定: 条件付き可）, PR #104
- 範囲: **要件定義のみ（実装なし）**
- 非対象: `/compare` 実装、OCRアルゴリズム変更、UI実装変更、payload仕様変更

## 0. 前提と目的

本ドキュメントは、`/card` OCR入力補助を限定導入するための安全条件を固定する。OCRは入力補助に限定し、最終確定はユーザー手動確認を必須とする。非公式ファンツールとして、公式素材利用可否は引き続き「要確認」とする。

## 1. `/card` の責務整理（既存）

`/card` は以下の責務を持つ。

1. 入力責務
   - 装備情報をフォーム入力で受け付ける（手動入力が正）。
2. 表示責務
   - 入力値をカード生成前にプレビュー表示する。
3. 保存責務
   - 既存保存payload仕様で状態を保存する。
4. 共有責務
   - 既存共有URL仕様へ変換し、互換性を維持する。
5. ランキング連携責務
   - 既存ランキングpayload仕様で投稿/連携する。

> OCR導入後も、上記3-5の責務境界は変更しない。

## 2. `/score` から流用する責務 / `/card` 固有責務

### 2.1 流用可能（`/score` と共通）

- OCRは画面内一時状態のみで扱う（永続化しない）。
- 低信頼度候補は自動確定しない（手動確認必須）。
- OCR失敗時の即時手動fallback導線を維持する。
- payload非混入方針（confidence/raw text/candidatesを保存しない）。
- 画像非保存（メモリ一時処理のみ）。

### 2.2 `/card` 固有として分離する責務

- カード生成前確認ステップの明示（生成物に直結するため）。
- 入力下書き反映後の「未確定項目」残存可視化。
- カード生成実行前に未確認項目がある場合の再確認導線。
- 共有・ランキングに影響させない回帰確認を`/card`個別で実施。

## 3. `/card` OCR入力補助 主ユースケース

1. 装備情報の下書き反映による入力時間短縮。
2. 複数項目のうち、認識できた項目のみ先行反映し、残りは手動補完。
3. 低信頼度/未確定項目を視認しながら安全に手動確定。
4. OCR失敗時にカード作成フローを中断せず、即手動入力へ復帰。

## 4. 項目別の反映ルール

### 4.1 OCRで下書き反映してよい項目

- 装備タイプ
- メインステータス名
- メインステータス値
- サブステータス名
- サブステータス値

### 4.2 手動確認が必須な項目

- confidence閾値未満の項目（閾値案: `< 0.85` を維持）
- マッピング一致タイプが弱い項目（partial/similar）
- 候補が複数競合して一意決定できない項目

### 4.3 未確定項目の扱い

- 未確定項目は空欄または「要確認」表示で残す。
- 自動確定しない。
- カード生成前確認で未確定数を明示する。

## 5. payload非混入方針（固定）

以下のOCRメタ情報は**保存payload・ランキングpayload・共有URLへ混入禁止**。

- confidence
- raw text
- matchType
- unresolvedReason
- 候補配列（candidates）
- OCR処理時間などの計測メタ

実装時は既存payload builderを利用し、OCR情報はUIの一時状態に閉じ込める。

## 6. 導線要件（確認・fallback・再試行）

1. カード生成前確認導線
   - OCR反映項目と未確定項目を生成実行前に確認できること。
2. 手動fallback導線
   - OCR失敗時に即座に手動入力へ戻れること。
3. 再試行導線
   - 読み取り結果が不十分な場合、同一セッションで再実行できること。
4. 低信頼度維持方針
   - 低信頼度項目は自動確定せず、常に確認待ち状態にする。

## 7. 監視KPI定義（`/card` 導入時）

## 7.1 必須KPI

- OCR処理時間 p95（全体）
- 端末性能別OCR処理時間 p95（低/中/高）
- 補正率（OCR反映後にユーザーが手動修正した割合）
- 未確定項目残存率（カード生成前時点）
- 反映前確認離脱率（確認導線で離脱した割合）
- fallback率（OCR失敗やユーザー判断で手動入力へ移行した割合）

## 7.2 監視判定の運用条件

- KPIは少なくとも1サイクルで日次集計、限定公開中は週次レビュー。
- 低スペック端末カテゴリを分離監視する。
- `/score` と同じ閾値を初期適用し、乖離がある場合は要確認として次Issueにエスカレーションする。

## 8. 回帰観点（互換性）

1. 保存payload非混入
   - OCR由来メタ情報が保存payloadに出現しないこと。
2. 共有URL互換
   - OCR有無でURL仕様差分が発生しないこと。
3. ランキング互換
   - OCR有無でランキングpayload仕様差分が発生しないこと。
4. 既存手動入力互換
   - OCR未使用時の従来導線・結果が変わらないこと。

## 9. 限定公開時ロールバック条件

以下のいずれかで即時ロールバック候補とする。

- KPIが2サイクル連続未達。
- payload混入、共有URL非互換、ランキング不整合の検知。
- 低スペック端末p95が悪化トレンド（例: 4.3秒超継続）。
- OCR誤反映起因の重大インシデント（誤誘導/誤投稿）が発生。

ロールバック手段はfeature flagで`/card` OCR導線を無効化し、手動入力のみへ戻す。

## 10. 次実装Issueの最小スコープ

1. `/card` にOCR入力補助UI導線を追加（限定公開）
2. カード生成前確認ステップを追加（未確定表示を含む）
3. 低信頼度自動確定禁止を実装で担保
4. payload非混入回帰テスト（保存/共有URL/ランキング）を追加
5. KPI計測イベントを`/card`向けに追加

## 11. ADR要否

**今回（Issue #105）ではADR不要**。

理由: 実装・恒久アーキテクチャ変更を確定しておらず、要件固定フェーズのため。

ただし、次Issueで`/card`限定導入を正式実装決定する時点で、段階展開・ロールバック戦略をADR化する。
