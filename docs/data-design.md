# データ設計方針

## 1. 全体方針
- 永続化を前提にDBを利用（Supabase想定）
- ゲスト利用時はローカル保存併用
- 設定データは更新容易性を重視

## 2. データ分類
1. マスタデータ
   - キャラ
   - ロール
   - ステータス
   - スコア重み
   - プリセット
2. トランザクションデータ
   - ユーザー
   - ビルド
   - ビルド履歴
   - ランキング集計
3. コンテンツデータ
   - お知らせ
   - FAQ
   - アップデート履歴

## 3. 保存方針
- ログインユーザー: サーバー保存
- ゲストユーザー: ローカル保存 + 任意で共有URL化
- 保存件数: 上限を設ける（具体値は要確認）

## 4. 共有方針
- 共有URLにより入力内容復元
- 共有データの有効期限・匿名化ルールは要件化

## 5. i18n方針
- 主要マスタは多言語カラムまたは辞書テーブルで管理
- 日本語を主軸として英語・韓国語を追加

## 6. セキュリティ方針
- 管理機能は権限分離
- 監査ログを考慮
- 個人情報最小化

## 7. 要確認事項
- 公式名称に合わせた項目定義
- OCR抽出データの正規化ルール
- ランキング不正対策

## 8. スコア設定データ設計（Issue #3 追記）
- `score_profiles`（ロール別/キャラ別/デフォルト）をマスタ管理する。
- `stat_ranges`（%/固定値別の min/max）をマスタ管理する。
- `rank_thresholds` はプロファイル単位で保持し、管理画面から更新可能にする。
- `custom_weight_presets` はユーザー単位で保存し、`base_profile_id` と差分（overrides）を保持する。
- `config_version` を各プロファイルに持たせ、計算時にバージョン整合チェックを実施する。

## 9. 要確認（データ設計）
- 実ゲームデータとの同期方法（手動更新 / 取込バッチ / API連携）
- セット効果・装備効果のID体系と一意制約
- カスタムプリセット保存上限件数
