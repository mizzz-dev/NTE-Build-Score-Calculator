# Issue #97 作業ログ（/score OCR処理時間 p95 未達改善）

- 日付: 2026-05-13
- 対象Issue: #97
- スコープ: `/score` OCR入力補助UX（待機中ガイダンス、事前画像ガイド、手動fallback導線）

## 1. 背景（第2サイクル未達）
- `docs/logs/ocr-quality-evaluation-cycle-2.md` で、OCR処理時間 p95 が `4.1秒` と基準（4.0秒以内）未達。
- 低スペック端末カテゴリの p95 `4.9秒` が全体p95を押し上げている。

## 2. 原因仮説
1. 低スペック端末ではOCR処理が4秒を超えやすいが、待機継続/切替判断のタイミングが弱い。
2. 高解像度・大容量画像やぼやけ画像で処理時間と再試行回数が増加しやすい。
3. 長時間処理時に fallback へ切り替える判断が利用者依存で、運用ばらつきが残る。

## 3. 実施内容
1. OCR処理中メッセージを経過時間段階（通常/4秒超/6秒超）で出し分け。
2. 「6秒以上」は継続待機と手動fallback/手動入力切替の判断を明示。
3. 読取前チェックとして、画像サイズ目安（2MB以下推奨）とぼやけ再撮影ガイドを追加。
4. ガイダンス関数の単体テストを追加し、閾値文言回帰を防止。

## 4. 維持した方針
- 低信頼度候補を自動確定しない。
- OCR結果・画像データを保存payload/ランキングpayload/共有URLへ含めない。
- `/card` `/compare` へ影響を与えない。

## 5. 次アクション
- 第3サイクル評価で、低スペック端末カテゴリ p95 の改善有無を再測定する。
- 4秒超過率と6秒超過率の計測観点を運用ログに追記検討する。
