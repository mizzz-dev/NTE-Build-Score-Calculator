# マスタデータ運用手順（Issue #63）

## 1. 目的とスコープ

本書は、公開マスタ由来の `ScoreConfig` を `/score` `/card` `/compare` に適用済みである前提で、
管理画面で扱う以下の運用データを安全に投入・更新・検証・切り戻しするための手順を定義する。

- キャラクター
- ロール
- ステータス
- スコア重み

本書の対象は**運用設計・手順・検証観点**に限定し、以下は本Issueの対象外とする。

- 実データの大量投入
- DB migration追加
- 管理画面CRUDの変更
- スコア計算式 / スコア重み仕様変更
- 保存済みデータ migration
- ランキング仕様変更
- OCR実装

## 3. 初期投入対象データ

### 3.1 ロール
- 用途: キャラクターに紐づく役割分類
- 参照先: キャラクターマスタ、スコア重み

### 3.2 ステータス
- 用途: スコア計算・表示で利用する評価軸
- 参照先: スコア重み

### 3.3 キャラクター
- 用途: `/score` `/card` `/compare` の選択対象
- 参照先: ロール

### 3.4 スコア重み
- 用途: `ScoreConfig` の重み定義
- 参照先: ロール、ステータス

## 4. マスタ別の入力要件（必須/任意/命名規則）

> 注意: 実際の列名は管理画面・既存スキーマに従うこと。本節は運用ルールの定義であり、スキーマ変更を伴わない。

### 4.1 ロール
- 必須項目
  - 一意キー（例: `role_key`）
  - 表示名
  - 有効フラグ
  - 表示順
- 任意項目
  - 説明文
  - 管理メモ
- 命名規則
  - 一意キーは英小文字 + ハイフン区切り（例: `attack`, `support`）
  - 表示名は公開画面の表記と一致
  - 表示順は10刻み（将来挿入余地の確保）

### 4.2 ステータス
- 必須項目
  - 一意キー（例: `status_key`）
  - 表示名
  - 単位/フォーマット区分
  - 有効フラグ
  - 表示順
- 任意項目
  - 小数桁数
  - 管理メモ
- 命名規則
  - 一意キーは英小文字 + スネークケース（例: `critical_rate`）
  - 表示名は `/score` `/card` `/compare` で統一
  - 単位は既存仕様の区分値のみ使用

### 4.3 キャラクター
- 必須項目
  - 一意キー（例: `character_key`）
  - 表示名
  - ロール参照
  - 有効フラグ
  - 表示順
- 任意項目
  - 略称
  - 実装/未実装フラグ
  - 管理メモ
- 命名規則
  - 一意キーは英小文字 + ハイフン区切り（例: `alice-a`）
  - 表示名は公開マスタの正式表記を優先
  - 略称を使う場合は重複禁止

### 4.4 スコア重み
- 必須項目
  - 対象ロール
  - 対象ステータス
  - 重み値
  - 有効開始日時
  - 有効フラグ
- 任意項目
  - 有効終了日時
  - 版番号/変更理由
  - 管理メモ
- 命名規則
  - 重み値は既存仕様の許容範囲内（0以上、上限は仕様準拠）
  - 同一ロール×ステータス×有効期間の重複を禁止
  - 版番号を運用する場合は `vYYYYMMDD-N` 形式を推奨

## 5. 管理画面投入 vs コード/seed管理の分離

### 5.1 管理画面で投入・更新するデータ
- 日常運用で変更が発生しうる値
  - キャラクターの有効/無効
  - 表示順の微調整
  - スコア重みの版切替（有効期間管理）

### 5.2 コード管理・seed管理すべきデータ
- 基準値として履歴管理が必要な値
  - ロール基本定義
  - ステータス基本定義
  - 初期版スコア重みテンプレート
- 理由
  - 差分レビュー可能
  - ロールバック容易
  - 環境間再現性確保

### 5.3 運用方針
- 本番反映は「seed相当の基準定義」+「管理画面の最終調整」で実施
- 管理画面のみで完結した変更は、当日中に差分記録（運用ログ）を必須化

## 6. 初期投入の推奨順序

1. ロール
2. ステータス
3. キャラクター
4. スコア重み
5. 公開画面での検証（`/score` `/card` `/compare`）

### 6.1 順序の理由
- 参照元（ロール/ステータス）を先に確定し、参照先（キャラクター/重み）を後投入することで不整合を防ぐ。

## 7. データ更新時の影響範囲

### 7.1 `/score`
- 入力候補（キャラクター/ロール）
- スコア計算結果（重み変更時）
- 共有URL復元時の表示差異（名称変更時）

### 7.2 `/card`
- 入力候補・表示文言
- 保存カード生成時の計算結果

### 7.3 `/compare`
- A/B両フォームの選択肢整合
- 同一入力でも重み更新により比較結果が変動

### 7.4 ランキング投稿
- 投稿時点の計算結果は重み更新の影響を受ける
- 古い投稿との単純比較が難しくなるため、版管理・注記が必要

### 7.5 保存済み履歴
- 既存履歴の payload 互換は維持前提
- マスタ名称変更により表示上の差異が起こりうる

### 7.6 共有URL
- URLスキーマを変更しない前提では復元可能
- ただし無効化されたキーは復元時に「要確認」扱いでフォールバック表示を確認する

## 8. 本番投入前チェックリスト

- [ ] 投入対象のCSV/入力一覧がレビュー済み
- [ ] ロールキー重複なし
- [ ] ステータスキー重複なし
- [ ] キャラクターキー重複なし
- [ ] 参照整合（キャラクター→ロール、重み→ロール/ステータス）OK
- [ ] 重みの欠損（主要ロール×主要ステータス）なし
- [ ] 同一期間の重み重複なし
- [ ] stagingで `/score` `/card` `/compare` の基本導線確認済み
- [ ] 互換性確認（保存・共有・ランキング）実施済み
- [ ] 切り戻しデータ（直前バックアップ）を確保
- [ ] 作業者/レビューア/承認者を記録

## 9. 更新後の回帰確認手順

1. `/score`
   - 代表キャラクター3件以上でスコア計算
   - 共有URL作成→別ブラウザ復元
2. `/card`
   - 同条件でカード生成
   - ゲスト履歴保存・クラウド保存（可能な環境）
3. `/compare`
   - A/B入力後に比較結果表示
   - 比較後もフォーム状態が保持されることを確認
4. ランキング投稿
   - 投稿処理成功
   - 投稿データに不要情報（token/secret等）が含まれない
5. ログ監視
   - 公開マスタ取得失敗時のfallbackが異常終了しない

## 10. 誤投入時の切り戻し方針

### 10.1 基本方針
- 変更前バックアップ（エクスポート or seedスナップショット）を必須化
- 切り戻しは「全量復元」ではなく「影響範囲限定の差分戻し」を優先

### 10.2 切り戻し手順
1. 障害申告（発生時刻・影響画面・対象キー）
2. 当該データを一時無効化（可能な場合）
3. 直前バックアップとの差分抽出
4. 差分反映（管理画面 or seed再適用）
5. `/score` `/card` `/compare` 最小回帰確認
6. 事後記録（原因・再発防止）

### 10.3 切り戻し判断基準
- 計算結果に有意な異常がある: 即時切り戻し
- 表示文言のみの軽微不整合: 営業時間内に計画修正

## 11. 権限別運用ルール（admin/editor/viewer）

### 11.1 admin
- 可能操作
  - 全マスタの作成/更新/無効化
  - 本番反映の最終承認
- 必須ルール
  - 本番更新は原則2名以上レビュー
  - 切り戻し可否判断と実行責任を持つ

### 11.2 editor
- 可能操作
  - 下書きデータ作成
  - staging相当環境での検証
- 制限
  - 本番反映の単独実行は禁止
  - ロール/ステータス定義の新設はadmin承認必須

### 11.3 viewer
- 可能操作
  - 閲覧、監査、確認
- 制限
  - 変更操作は禁止
