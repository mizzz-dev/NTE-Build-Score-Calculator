# /score OCR品質評価運用（MVP）

最終更新: 2026-05-13
対象Issue: #81

## 1. 目的
`/score` OCR入力補助MVPの品質を継続評価し、以下を定量的に判断できる状態を作る。
- 実運用で許容できる認識精度・補正負荷か
- 端末差を含めた処理時間がUX上許容範囲か
- OCR失敗時のfallback導線が機能しているか
- `/card`・`/compare` へ展開検討に進めるか

## 2. 非目的
- OCRアルゴリズム改善そのもの（モデル変更・辞書最適化等）
- `/card`・`/compare` への実装展開
- 画像保存基盤の追加、外部OCR API連携
- 保存payload仕様・DBスキーマ変更

## 3. 評価対象範囲
- 対象画面: `/score` のOCR入力補助（画像選択〜候補反映〜手動補正）
- 対象外: `/card` `/compare`、ランキング、管理画面CRUD、ダメージシミュレーター
- 前提PR: #66, #68, #70, #72, #74, #76, #78, #80

## 4. 評価指標（KPI）

### 4.1 ステータス名認識成功率
- 定義: 期待するステータス名文字列をOCRが正しく抽出できた件数 / 対象件数
- 単位: %

### 4.2 ステータス値認識成功率
- 定義: 期待するステータス値（数値）をOCRが正しく抽出できた件数 / 対象件数
- 単位: %

### 4.3 ステータスキー自動マッピング成功率
- 定義: OCR抽出結果が正しい内部ステータスキーへ自動マッピングされた件数 / 対象件数
- 単位: %

### 4.4 手動補正率
- 定義: OCR結果に対しユーザーが1項目以上手動修正したセッション数 / OCR実行セッション数
- 単位: %

### 4.5 OCR処理時間
- 定義: OCR実行開始から候補提示完了までの時間
- 集計: p50 / p95 / max（端末カテゴリ別）
- 単位: 秒

### 4.6 OCR失敗時fallback率
- 定義: OCR失敗時に手動入力へ遷移し、最終的に入力完了できた件数 / OCR失敗件数
- 単位: %

### 4.7 低信頼度候補の手動補正誘導率
- 定義: 低信頼度警告が出た項目のうち、ユーザーが手動確認・修正まで到達した件数 / 低信頼度警告件数
- 単位: %

## 5. サンプル画像カテゴリ
評価時は以下カテゴリを均等に含める。
- 端末種別: スマホスクショ / PCスクショ
- 解像度: 高解像度 / 低解像度
- 背景条件: 明るい背景 / 暗い背景
- 表示言語: 日本語 / 英語 / 韓国語
- 画質劣化: 圧縮あり / ぼやけあり

> 要確認: 実運用サンプル母数（1サイクルあたり最低件数）は運用開始後2週間の実測を踏まえて固定する。

## 6. 評価手順
1. 評価用画像セットをローカル端末で準備する。
2. 画像に個人情報・アカウント情報が含まれないことを事前確認する。
3. `/score` OCR入力補助で読取を実行し、期待値との差分を記録する。
4. OCR失敗時は手動入力fallbackへ遷移し、完了可否を記録する。
5. 記録は匿名・集計値のみ保持し、画像そのものは保存しない。
6. 週次でKPIを集計し、前週差分と端末差分を確認する。

### 6.1 データ取り扱い方針
- 画像はサーバー保存しない（メモリ上の一時処理のみ）。
- 個人情報・アカウント情報を含む画像は評価対象外。
- 評価結果は匿名化した集計値のみ記録し、原画像と紐づけない。

## 7. 受け入れ基準（暫定）
- ステータス名認識成功率: 95%以上
- ステータス値認識成功率: 98%以上
- ステータスキー自動マッピング成功率: 95%以上
- 手動補正率: 20%以下
- OCR処理時間: p95 が 4.0秒以内
- OCR失敗時fallback率: 99%以上
- 低信頼度候補の手動補正誘導率: 90%以上

> 要確認: 受け入れ基準は端末分布とサンプル母数の安定後に正式化する（本書はMVP暫定値）。

## 8. 失敗時の分類基準

### 8.1 OCR読み取り失敗
- 文字抽出そのものが不成立、または結果が空

### 8.2 マッピング失敗
- 文字抽出は成功したが内部ステータスキー紐付けに失敗

### 8.3 UI誘導不足
- 低信頼度・失敗時にユーザーが次アクションを理解できず離脱

### 8.4 端末性能問題
- 端末依存で処理時間が閾値を継続超過、または実行失敗率が高い

## 9. `/card`・`/compare` 展開判定基準
次の全条件を連続2サイクル（週次）で満たした場合に、要件再評価フェーズへ進む。
1. 7章の受け入れ基準をすべて達成
2. 失敗分類で「UI誘導不足」「端末性能問題」が増加トレンドでない
3. fallback完了率が高水準（99%以上）を維持
4. 重大インシデント（誤確定の大量発生）がない

満たさない場合は `/score` 改善を継続し、`/card` `/compare` 展開は保留する。

## 10. 運用ログ更新ルール
- 週次集計のサマリを `docs/logs/` に追記する。
- 閾値未達の場合は失敗分類ごとに原因仮説と次アクションを明記する。
- 関連Issueへ集計結果を紐づけ、ドキュメントのみ先行して実装を仮置きしない。
