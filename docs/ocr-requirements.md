# OCR要件定義（Issue #65）

## 1. 目的
本ドキュメントは、NTE Build Score Calculator にOCR機能を追加する前段として、要件・技術検証方針・既存機能との接続方針を整理する。

- 対象: 要件定義・設計・技術検証観点の整理
- 非対象: OCR実装、画像アップロード実装、外部OCR API連携、DB migration、保存payload仕様変更
- 前提PR: #52, #54, #56, #58, #60, #62, #64

## 2. 対象ユースケース
OCRは「入力補助」に限定し、最終確定値はユーザーが確認・補正する。

1. スコア計算への入力補助（`/score`）
2. ビルドカード生成への入力補助（`/card`）
3. 比較画面への入力補助（`/compare`）

## 3. 初期対象画面と段階導入
### 3.1 初期対象（Phase 1）
- `/score` の入力補助のみを初期対象とする。
- 理由:
  - 既存の入力モデルが単純で補正UIを最初に検証しやすい。
  - 保存・共有・ランキング互換（PR #62）への影響範囲を局所化できる。

### 3.2 対象外（Phase 1時点）
- `/card` のOCR導線
- `/compare` のOCR導線
- 画像の永続保存
- OCR結果の自動確定（確認なし反映）

### 3.3 後続拡張（Phase 2以降）
- `/card`, `/compare` への展開は、`/score` で精度・UX基準を満たした後に実施する。

## 4. 読み取り対象項目
## 4.1 初期対象項目
- 装備タイプ
- メインステータス名
- メインステータス値
- サブステータス名
- サブステータス値

## 4.2 初期対象外（要確認）
- キャラクター名
- ロール

理由: キャラクター/ロールは公開マスタとの曖昧一致コストが高く、誤認時の重み適用ミスがスコア差分に直結するため、初期は手動選択を維持する。

## 5. 入力画像条件
## 5.1 形式
- 対応候補: `image/png`, `image/jpeg`, `image/webp`
- 非対応: HEIC/RAW/PDF（初期）

## 5.2 サイズと解像度
- 最大ファイルサイズ: 5MB（初期案）
- 推奨解像度: 短辺 1080px 以上
- 下限解像度（警告表示）: 短辺 720px 未満は精度低下警告

## 5.3 端末別導線
- PC: ファイル選択（ドラッグ&ドロップは後続検討）
- モバイル: カメラロール選択（直接撮影導線は後続検討）

## 5.4 画像保存方針
- サーバー保存しない（メモリ上の一時処理のみ）
- ローカル永続保存しない（ユーザー操作で再選択前提）

## 6. セキュリティ/プライバシー方針
1. 画像データはサーバーへ永続保存しない。
2. OCR結果payloadに不要情報を含めない。
3. access token / secret / service role key をOCR経路で扱わない。
4. 画像に個人情報・アカウント情報が含まれる可能性を事前警告する。
5. ログ出力は最小化し、OCR生画像や秘匿文字列を記録しない。

### 6.1 注意表示文言（案）
- 「スクリーンショット内にUID・メールアドレス等の個人情報が含まれないことを確認してください。」
- 「OCR結果は自動入力の補助機能です。必ず内容を確認して保存してください。」

## 7. OCR方式候補比較
| 方式 | 概要 | 長所 | 短所/リスク | 初期採用判断 |
|---|---|---|---|---|
| ブラウザ内OCR | クライアントで文字認識を実行 | 画像を外部送信しない設計と整合、低レイテンシ | 端末性能依存、実装/チューニングコスト | 第一候補 |
| サーバー/API OCR | サーバーまたは外部APIで認識 | 精度向上余地、端末差吸収 | 外部送信・コスト・法務/プライバシー負荷増 | 初期不採用 |
| 手動入力補助のみ | OCRなしで入力支援UIのみ改善 | 実装リスク最小、即時提供可能 | OCR期待に未達 | 暫定fallbackとして併用 |

### 7.1 採用基準（初期）
- セキュリティ方針（画像非保存）を満たすこと
- `/score` 入力時間短縮に寄与すること
- 手動補正フローで誤認識を確実に修正可能であること

## 8. 技術検証観点
1. 多言語認識: 日本語/英語/韓国語のステータス名認識率
2. 数値認識: 整数・小数・`%` の認識率
3. 正規化: 全角半角、記号揺れ、`crit rate` vs `critical rate` 等の吸収
4. 公開マスタマッピング: 認識語をステータスキーへ一意または候補付きで変換
5. 低精度時UX: 信頼度低い項目の警告表示と手動補正導線

### 8.1 判定基準（案）
- 必須項目入力完了率: 95%以上（補正込み）
- 主要ステータス名一致率: 90%以上
- 数値項目一致率: 95%以上
- OCR利用時でも最終保存payload互換が維持されること

## 9. OCR結果データ構造案
保存payload仕様は変更しないため、OCR結果は画面内の一時状態として扱う。

```ts
type OcrFieldConfidence = {
  value: string;
  confidence: number; // 0.0 - 1.0
  normalizedValue?: string;
  candidates?: string[];
};

type OcrEquipmentDraft = {
  equipmentType?: OcrFieldConfidence;
  mainStatName?: OcrFieldConfidence;
  mainStatValue?: OcrFieldConfidence;
  subStats: Array<{
    statName?: OcrFieldConfidence;
    statValue?: OcrFieldConfidence;
  }>;
  warnings: string[];
  sourceLocale?: 'ja' | 'en' | 'ko' | 'unknown';
};
```

- `OcrEquipmentDraft` はUI補助専用。
- 保存時は既存フォーム値へ確定反映し、既存payloadBuilderをそのまま利用する。

## 10. 既存設計との接続方針
## 10.1 公開マスタ（PR #52, #54, #64）
- ステータス名の同義語・表記揺れ吸収は公開マスタ由来キーにマッピングする。
- マッピング失敗時は「未確定項目」として手動選択へフォールバックする。

## 10.2 ScoreConfigFactory / ScoreConfig（PR #56, #58, #60）
- OCRは「入力値補助」層に限定し、ScoreConfig生成ロジックには介入しない。
- `/score` で確定した値は既存ScoreConfigで通常計算する。

## 10.3 ScoreShareState / payloadBuilders / ランキング（PR #62）
- OCR由来メタ情報（confidence, raw text）は保存payloadに含めない。
- 保存・共有・ランキング投稿は既存payloadBuilderを再利用し、互換性を維持する。

## 11. fallback・エラー表示・手動補正方針
1. 画像読取失敗: 「読み取りに失敗しました。手動入力してください。」
2. 部分失敗: 認識できた項目のみ候補入力し、未確定は空欄維持
3. 低信頼度: 該当入力を警告色表示し、候補選択UIを提示
4. マッピング失敗: 公開マスタ候補一覧から手動選択
5. 最終保存前: OCR反映項目の確認ステップを必須化（初期）

## 12. 実装PR分割案
1. **PR-A（設計/型定義）**
   - OCR要件反映、画面内一時データ型追加（永続化なし）
2. **PR-B（`/score` OCR UI骨組み）**
   - 画像選択UI、注意表示、手動補正UI、fallback文言
3. **PR-C（OCRエンジン接続）**
   - ブラウザ内OCR接続、正規化、マッピング
4. **PR-D（品質強化）**
   - 多言語検証、精度改善、回帰テスト整備
5. **PR-E（横展開判断）**
   - `/card` `/compare` への展開可否判定（要件レビュー）

## 13. リスク・要確認事項
- 要確認: NTE画面仕様変更時のOCR耐性
- 要確認: 韓国語フォント・数字表記差異の認識品質
- 要確認: 低スペック端末での処理時間上限
- 要確認: 公式素材利用規約との整合（非公式ファンツール表記維持）

## 15. OCRマッピング精度改善方針（Issue #77）
### 15.1 ステータス名正規化
- 全角/半角は `NFKC` 正規化で吸収する。
- 大文字/小文字は小文字化して比較する。
- 空白・記号は比較用トークンで除去/統一する。
- `%` と「率」は比較時に同一カテゴリとして扱う。
- 「ダメージ / dmg」「critical / crit / 会心」「attack / atk / 攻撃力」「defense / def / 防御力」「hp / 体力」を同義語正規化する。
- 韓国語表記（例: `치명타`, `공격력`）は比較トークン化で将来拡張可能な構造を維持する。

### 15.2 公開マスタ候補生成
- 候補は以下の優先順で評価する。
  1. 完全一致（exact）
  2. 正規化一致（normalized）
  3. alias一致（alias）
  4. 部分一致（partial）
  5. 類似一致（similar）
- 上位3件まで候補を保持し、UIで確認可能にする。
- 公開マスタに alias が未整備でも `displayName` 比較で安全にfallbackする。

### 15.3 信頼度と手動補正
- 各候補に `confidence`（0.0-1.0）と `matchType` を付与する。
- しきい値未満（`< 0.85`）は自動確定せず、未解決理由を明示して手動補正必須とする。
- 手動補正完了後のみ既存フォームに反映できる状態にする。

### 15.4 payload非混入保証
- OCR由来メタ情報（`confidence`, `matchType`, `unresolvedReason`, 候補一覧）は画面内一時状態に限定する。
- 保存payload・ランキングpayload・共有URLにはOCRメタ情報を含めない。
